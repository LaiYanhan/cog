# 问题与论点

> 本文阐述 cog 解决的问题。这是整个项目的**起点**，回答"为什么要造这个工具"。理论细节见 [02-cognitive-model.md](02-cognitive-model.md)，价值定位见 [03-latent-space.md](03-latent-space.md)。

## 1 LLM 代码的腐败问题

人类开发者编写的代码具有一种抽象的**稳定性**：开发者对代码存在唯一且稳定的认知——至少能够回忆起"当时为什么在这里用这种方式写了代码"。这种认知模型潜藏在人脑中，支撑着代码的长期维护。

LLM 生成的代码缺乏这种稳定性。同一个 LLM 在上一个 session 中写下的代码，下一个 session 可能产生完全不同的解释。这种认知不连续导致：

- **短视修补**：遇到报错时不去溯源根因，而在表象层面打补丁。
- **认知漂移**：对同一代码块的"理解"随 session 变化，缺乏一致性。
- **腐败加速**：每次修改都可能引入与先前决策意图矛盾的改变。
- **归因缺失**：无法回答"为什么这里是这样而不是那样"。

## 2 问题本质

LLM 的根本缺陷不是代码质量，而是**没有持久的外部认知模型**。人类维护代码时依赖大脑中隐含的认知结构——因果链、不变量、设计意图、修正历史——而 LLM 每次交互都从零开始构建理解。

> **核心论点**：为 LLM 构建一个外部化的认知模型——可构建、可追溯、可更新、且唯一——可以从根本上缓解 LLM 代码的腐败问题。

## 3 为什么不用评分/树搜索方案

一个自然的替代方案是"树搜索 + 评分"：生成多个候选实现，用分数引导搜索。但它在 coding agent 场景下不可行，原因在于**不存在可靠的自动化连续评分信号**：

- Agent 自己写的测试 → 几乎必然满分（天花板效应，信号方差 ≈ 0）。
- LSP 诊断 → 跨轮次的中间状态无意义，最终状态几乎必然无错。
- LLM 自评 → 不独立，自我肯定偏差。
- 项目已有测试 → 唯一可靠的外部信号，但多数项目不充分。

这是信息论约束，不是调参能解决的问题。

cog 绕过这个约束的方式：用证伪（二元信号）替代评分（连续信号），用解释（哪条假设错了）替代分数（多高多低）。模型不判断代码"好坏"，只忠实记录推理链，并在推理被推翻时沿依赖边确定性地传播影响。

## 4 cog 是什么，不是什么

明确定位边界，避免误用：

- **是推理持久化工具**——存的是推理链（intent、fragility、correction、设计决策），不是事实清单。断言携带因果结构，读到它就不必重新理解"为什么这样改过"。
- **是影响面计算器**——把"如果改 Store 会影响什么"从逐文件阅读的推理问题，变成 `cog impact` 的一次 BFS 遍历。
- **不是静态分析器**——行数、复杂度、相似度是 SonarQube 的领地。
- **不是代码索引**——ctags/LSP 已经做得足够好。cog 的 entity 列表是推理的锚点，不是代码浏览的替代。
- **不是文档生成器**——断言是给 agent 的推理输入，不是给人读的 API 文档。

## 5 边界：cog 解决不了什么

忠实记录当前认知的局限（见 [03-latent-space.md](03-latent-space.md) §4 详述）：

- **实体元数据稀疏**：tree-sitter 扫描只提取名称、kind、行数、fan-in/out。一个 200 行的重构方法和一个 15 行的简单查询在结构信号上差异有限。
- **结构依赖图不捕获运行时语义**：generator protocol、并发约束、框架级隐式依赖都不在 entity 关系图中。这正是 [下降协议](../concepts/02-descent-protocol.md) 要应对的 fidelity gap。
- **断言间 `depends_on` 实践中使用率低**：TMS 级联的威力在真实使用中尚未充分发挥，因为 agent 极少主动建立断言间依赖链。

这些不是要隐藏的缺陷，而是 cog 作为"结构化记忆"而非"全知预言机"的诚实定位。
